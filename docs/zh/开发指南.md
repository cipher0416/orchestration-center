# 编排中心开发指南

## 1. 特性介绍
编排中心是一个面向多智能体（Agent）协作的可视化编排平台，支持通过图形化工作流设计器定义 Agent 之间的调用关系与执行流程。后端基于 Python 框架解析流程并驱动 Agent 协同工作，帮助用户高效构建、管理和运行复杂的 Agent 协作流程。

本特性允许用户为不同操作定义自定义实现，同时为常见操作提供默认实现。通过统一的抽象基类和处理器注册机制，开发者可以灵活扩展系统功能，无需修改核心代码。
除上述自定义实现以外本特性还提供了一个可扩展的大模型（LLM）接入框架，支持用户自定义实现不同厂商或类型的大模型。框架基于注册器模式，可以方便地添加新的大模型类型并在代码中使用。

## 2. 约束与限制
- **操作限制**：
  - 自定义处理器注册时应在应用启动时完成，避免在业务处理过程中注册
  - 自定义LLM实现应在应用启动前完成，避免应用启动后完成导致找不到自定义的LLM

## 3. 环境要求
- Python >= 3.10+

## 4. 使用场景
### 4.1 自定义处理器使用说明
#### 4.1.1 功能介绍
**核心能力：**
- **抽象基类**：为所有处理器定义统一接口
- **默认实现**：为常见操作提供内置处理器
- **自定义扩展**：支持用户注册自定义处理器

**系统架构：**
系统采用注册表模式，核心组件包括：

| 组件 | 职责 |
|------|------|
| BaseHandle | 定义所有处理器的统一抽象接口 |
| HandlerRegistry | 管理处理器注册和获取，提供默认实现兜底 |
| InterfaceType | 定义支持的接口类型枚举 |
| 默认处理器 | 为常见操作提供内置实现 |
| 自定义处理器 | 用户扩展实现，覆盖默认行为 |

```
┌─────────────────┐
│   业务调用方     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ HandlerRegistry │◄──── 注册自定义处理器
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   BaseHandle    │
│   (抽象基类)     │
└────────┬────────┘
     ┌───┴───┬───────────────┐
     ▼       ▼               ▼
┌───────┐ ┌───────┐  ┌───────────┐
│ 默认  │  │ 默认 │   │  自定义   │
│处理器1│  │处理器2│  │  处理器3  │
└───────┘ └───────┘  └───────────┘
```
#### 4.1.2 开发步骤
步骤1：引入所需模块
```python
from default_handle import BaseHandle, HandlerRegistry, InterfaceType
```

步骤2：创建自定义处理器

创建继承自BaseHandle的自定义类，并实现handle方法：
```python
class MyCustomHandle(BaseHandle):
    """自定义处理器示例"""
    
    def handle(self, *args, **kwargs):
        """
        自定义处理逻辑
        
        Args:
            *args: 位置参数
            **kwargs: 关键字参数
            
        Returns:
            处理结果
        """
        # 自定义实现
        # 建议：在这里添加具体的业务逻辑
        # 注意：确保方法是异步的
        result = "自定义结果"
        return result
```
步骤3：注册自定义处理器

在应用启动时注册自定义处理器：
```python
# 注册自定义处理器
# 注意：注册操作应在业务处理开始前完成
HandlerRegistry.register(InterfaceType.QUERY, MyCustomHandle)
```
警告：后注册的处理器会覆盖同类型的先前处理器。请确保注册顺序符合预期。

步骤4：使用处理器

在业务代码中获取并使用处理器：
```python
# 获取处理器实例
# HandlerRegistry会自动返回已注册的自定义处理器或默认实现
handle = HandlerRegistry.get_handler(InterfaceType.QUERY)

# 使用处理器执行业务
result = handle.handle(...)
```
步骤5：完整示例

以下是一个完整的自定义处理器实现示例：
```python
from default_handle import BaseHandle, HandlerRegistry, InterfaceType

class SaveCustomHandle(BaseHandle):
    """自定义查询处理器"""
    
    def handle(self, query_params=None):
        """
        自定义查询逻辑
        
        Args:
            query_params: 查询参数字典
            
        Returns:
            查询结果列表
        """
        # 示例：实现自定义查询逻辑
        if not query_params:
            return []
        
        # 自定义查询实现
        results = []
        # ... 业务逻辑
        return results

# 注册
HandlerRegistry.register(InterfaceType.SAVE_PSOP, SaveCustomHandle)

# 使用
handler = HandlerRegistry.get_handler(InterfaceType.SAVE_PSOP)
data = handler.handle({"id": 1})
print(data)
```
#### 4.1.3 默认处理器说明

若未注册自定义处理器，系统将使用以下默认实现：

| 处理器 | 对应接口类型 | 功能描述 |
|--------|-------------|----------|
| SavePsopHandler | SAVE | 保存psop |
| GetAllPsopsHandler | QUERY_ALL | 查询所有psop |
| GetPsopHandler | QUERY | 通过psop的id查询对应的psop |
| DeletePsopHandler | DELETE | 删除psop |

### 4.2 自定义LLM使用说明
#### 4.2.1 功能介绍
**核心能力：**
- **抽象基类**：为所有LLM实现定义统一接口（`BaseLLM`）
- **注册器机制**：通过装饰器模式管理LLM提供者的注册与获取
- **配置驱动**：基于JSON配置文件管理不同LLM的连接参数
- **实例缓存**：单例模式管理LLM实例，避免重复创建

**系统架构：**
系统采用注册器模式与装饰器模式，核心组件包括：

| 组件 | 职责 |
|------|------|
| BaseLLM | 定义所有LLM的抽象基类，提供统一的调用接口 |
| LLMProviderRegistry | 管理LLM提供者注册，支持装饰器注册 |
| LLMType | 定义支持的LLM类型枚举 |
| LLMConfig/LLMConfigItem | 管理LLM配置信息（模型、API、密钥等） |
| 默认LLM实现 | OpenAIStyleLLM等内置实现 |
| 自定义LLM | 用户扩展实现，支持任意厂商接入 |

```
┌─────────────────────┐
│     业务调用方       │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   get_llm_instance  │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│ LLMProviderRegistry │◄──── @registry_provider 注册自定义LLM
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│      BaseLLM        │
│     (抽象基类)       │
└──────────┬──────────┘
     ┌──────┴──────┬───────────────┐
     ▼             ▼               ▼
┌─────────┐ ┌──────────┐  ┌─────────────┐
│OpenAI   │ │ AOC      │  │  自定义     │
│StyleLLM │ │ ChatLLM  │  │  MyLLM      │
└─────────┘ └──────────┘  └─────────────┘
```

#### 4.2.2 开发步骤
步骤 1：修改配置文件 `llm_config.json`

在配置文件中添加新的大模型配置项：

```json

    "model": "gpt-4",
    "api": "https://api.openai.com/v1",
    "apikey": "your-openai-api-key",
    "enable_thinking": false
  },
  "my_custom_llm": {
    "model": "custom-model-name",
    "api": "https://your-api-endpoint.com/v1",
    "apikey": "your-custom-api-key",
    "enable_thinking": true
  }
}
```
步骤 2：在 LLMType 枚举中添加新类型
```python
from enum import Enum

class LLMType(Enum):
    OPENAI_STYLE_LLM = "openai_style_llm"
    MY_CUSTOM_LLM = "my_custom_llm"  # 新增自定义类型
```
步骤 3：创建自定义大模型实现文件
```python
from typing import Union, Tuple
from common.llm.config.llm_config import LLMType, LLMConfig
from common.llm.provider.base_llm import BaseLLM
from common.llm.provider.llm_provider_registry import registry_provider

@registry_provider(LLMType.MY_CUSTOM_LLM)
class MyCustomLLM(BaseLLM):
    
    def __init__(self, llm_config: LLMConfig):
        super().__init__(llm_config)
        # 初始化自定义客户端
        self.client = {} # 按实际情况实现
    def _ask_llm(self, prompt: str) -> Union[str, Tuple[str, str]]:
        # 调用自定义API, 仅作为示例说明，具体实现按实际情况进行修改
        response = self.client.generate(
            model=self.model,
            prompt=prompt,
            enable_thinking=self.enable_thinking
        )
        
        # 根据API返回格式处理
        if self.enable_thinking and hasattr(response, 'reasoning'):
            reasoning = response.reasoning
            answer = response.content
            return reasoning, answer
        else:
            return response.content
```
步骤 4：在 common/llm/__init__.py 中导出新类
```python
from .llm import get_llm_instance
from .provider.llm_openai import OpenAIStyleLLM
from .provider.llm_openai import MyCustomLLM

           "MyCustomLLM"
           ]
```

## 5. 接口定义

> 所有内部 UI 端点前缀为 `/rest/v1/orchestrate`。外部系统集成端点前缀为 `/api/v1`（参见 [External Integration Guide](../external-api-guide.md)）。

### 5.1 PDF 解析接口

#### `POST /rest/v1/orchestrate/parse-pdf`

上传 PDF 文件并解析指定章节。

**请求**:

- **方法**: POST
- **Content-Type**: multipart/form-data

| 参数名 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| file | file | 是 | PDF 文件 |

**成功响应**:
```json
{
  "code": 200,
  "status": "success",
  "data": {
    "id": "preflow-uuid",
    "name": "filename.pdf",
    "description": "Workflow parsed from PDF filename.pdf",
    "created_at": "2026-05-20T21:00:00",
    "steps_md": "Markdown格式的步骤描述",
    "tags": []
  }
}
```
**错误响应**:

- 400: 未提供文件、文件名为空、非 PDF 文件，或未找到指定章节
- 413: 文件过大（超过 100 MB）
- 503: 服务繁忙
- 500: 解析失败

  "preflow": {

| `agent_cards` | List[object] | 是 | AgentCard 列表 JSON |

**成功响应**:
```json
{
  "code": 200,
  "status": "success",
  "data": { /* PSOP 工作流 JSON 数据 */ }
}
```

### 5.3 PSOP 管理接口

#### `GET /rest/v1/orchestrate/workflows`

获取工作流列表。

| 查询参数 | 类型 | 必填 | 默认值 | 描述 |
| --- | --- | --- | --- | --- |
| limit | integer | 否 | 10 | 返回结果数量限制 (1–100) |

**成功响应**:
```json
{
  "code": 200,
  "status": "success",
  "data": [ /* PSOP 对象列表 */ ]
}
```

#### `GET /rest/v1/orchestrate/workflows/{workflow_id}`

根据 ID 获取工作流详情。

**成功响应**:
```json
{
  "code": 200,
  "status": "success",
  "data": { /* 完整 PSOP 对象 */ }
}
```

**错误响应**: 404 — 工作流不存在

#### `POST /rest/v1/orchestrate/workflows`

保存工作流。HTTP 状态码 201。

**请求体**:

```json
{
  "psop": { /* PSOP 对象 JSON，必须符合 PSOP 模型定义 */ }
}
```

**PSOP 模型字段**:

| 字段 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| id | string | 否 | 自动生成 UUID |
| name | string | 是 | 工作流名称 |
| description | string | 否 | 工作流描述 |
| created_at | datetime | 否 | 创建时间（自动生成） |
| steps | List[Step] | 是 | 步骤列表 |
| related_preflow | string | 否 | 关联的 PreFlow ID |
| user_intent | string | 否 | 生成该工作流的原始用户意图 |
| tags | List[string] | 否 | 标签 |

**Step 模型字段**:

| 字段 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| name | string | 是 | 步骤标识 |
| type | StepType | 否 | 成功条件: `AllSuccess`（全部子任务成功）或 `AnySuccess`（任一子任务成功），默认 `AllSuccess` |
| subtasks | List[Task] | 是 | 子任务列表，子任务间无依赖可并行执行 |
| next | List[JumpCondition] | 否 | 跳转条件列表，为空则无条件跳转 |
| layer | int | 否 | 编排层级: 0 = 执行层（叶子Agent），1+ = 聚合层，默认 0 |
| context_from | List[string] | 否 | 提供上下文的步骤名称列表，`["*"]` 表示所有已执行步骤 |

**Task 模型字段**:

| 字段 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| task_id | string | 否 | 自动生成 UUID |
| description | string | 是 | 任务描述 |
| agent | string | 是 | 执行任务的 Agent 名称 |
| skill | string | 是 | 执行任务所需的技能 |
| status | TaskStatus | 否 | 任务状态，默认 `pending`。枚举值: `pending`, `running`, `success`, `failed` |

**成功响应**:
```json
{
  "code": 201,
  "message": "Workflow created successfully",
  "status": "success",
  "data": { "workflow_id": "uuid" }
}
```

#### `DELETE /rest/v1/orchestrate/workflows/{workflow_id}`

删除指定 ID 的工作流。

**成功响应**:
```json
{
  "code": 200,
  "message": "Workflow {workflow_id} deleted",
  "status": "success"
}
```

**错误响应**: 404 — 工作流不存在

> **兼容性说明**: 系统同时保留了旧版路径别名（无前缀）: `GET /psops`、`GET /psops/{id}`、`POST /psops`、`DELETE /psops/{id}`，功能与新路径一致，建议优先使用带前缀的新路径。

### 5.6 SSE 执行接口

#### `GET /rest/v1/orchestrate/execute`

启动 PSOP 执行并通过 SSE 实时推送进度。

| 查询参数 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| psop_id | string | 是 | PSOP 工作流 ID |
| user_intent | string | 否 | 运行时用户意图和场景描述。提供时，引擎将该上下文注入到**所有步骤**（包括 layer-0 叶子步骤），使 Agent 在执行预定义任务的同时获得用户的原始场景数据。对于通过意图检索匹配到的 PSOP，强烈建议传入此参数 |

**重要说明**: `user_intent` 参数用于在执行检索到的 PSOP 时传入用户的原始意图。PSOP 的预定义任务描述（如"检查服务器电源和环境状态"）不含具体场景数据（如"华为园区早上8点出现网络故障"）。传入 `user_intent` 后，引擎在构建每个步骤的上下文时会自动注入 `## Runtime Context` 段，包含用户的原始意图描述。

**事件类型**: init, start, agent_request, agent_response, psop_update, complete, error, close

**psop_update 事件**: 推送完整的 PSOP 对象（含更新的任务状态），不是精简摘要。`data.psop` 字段包含完整 PSOP JSON。

**curl 示例**:

```bash
# 基本执行
curl -N "http://127.0.0.1:60000/rest/v1/orchestrate/execute?psop_id=6a204d60-f2ff-471b-9892-c5beae1c3a5c"

# 带运行时意图的执行（推荐）
curl -N "http://127.0.0.1:60000/rest/v1/orchestrate/execute?psop_id=6a204d60-f2ff-471b-9892-c5beae1c3a5c&user_intent=SPN%E4%B8%93%E7%BA%BF%E6%95%85%E9%9C%9C%E8%AF%81%E6%96%AD"
```

## 6. FAQ
### 6.1 自定义处理器未生效
**现象描述**：注册了自定义处理器，但调用时仍使用默认处理器。

**可能原因**：
- 1.注册操作在获取处理器之后执行
- 2.注册的接口类型与实际使用的不匹配
- 3.自定义处理器未正确继承BaseHandle

**解决办法**：
- 1.确保在应用启动时、任何业务处理之前完成注册
- 2.检查注册和获取时使用的InterfaceType是否一致
- 3.确认自定义处理器正确继承了BaseHandle并实现了handle方法

### 6.2 新添加的大模型没有被注册怎么办？
**现象描述**：已经按照文档添加了自定义大模型，但在调用时提示找不到该类型，或仍然使用默认的大模型。

**可能原因**：
- 1.未正确使用 @registry_provider 装饰器
- 2.自定义大模型类未在 __init__.py 中被导入
- 3.LLMType 枚举的字符串值与配置文件中的 key 不一致
- 4.自定义大模型文件未被 Python 解释器加载 

**解决办法**：
- 1.确认在自定义类定义处使用了 @registry_provider(LLMType.YOUR_TYPE) 装饰器
- 2.在 llm/__init__.py 中添加导入语句：from .your_custom_llm import YourCustomLLM
- 3.检查 LLMType 枚举值与 llm_config.json 中的配置键名是否完全一致（包括大小写）
- 4.确保自定义大模型模块在应用启动时被导入，可在 __init__.py 中导入，或在主程序入口导入

### 6.3 PDF 解析失败，提示未找到指定章节

**现象描述**：上传 PDF 文件后，返回错误“PDF解析失败，未找到指定章节”。

**可能原因**：PDF 文档中不包含标题为 "5. Interaction Flow" 的章节。

**解决办法**：确保上传的 PDF 文档包含该章节，或联系管理员扩展支持的章节类型。

### 6.4 执行工作流时没有实时进度推送

**现象描述**：点击执行按钮后，右侧区域没有显示执行过程。

**可能原因**：

- 后端 SSE 连接异常
- 浏览器不支持 EventSource
- 工作流中的 Agent 服务未启动

**解决办法**：

1. 打开浏览器开发者工具，查看 Network 面板中 SSE 连接是否正常
2. 确认 Agent 服务已启动且网络可达
3. 尝试使用最新版 Chrome/Firefox 浏览器
